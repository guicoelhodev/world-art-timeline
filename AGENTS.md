# Repository Instructions

## Current State

- This is a fresh SvelteKit 2 + Svelte 5 app generated by `sv`; product code is not implemented yet.
- The product plan lives in `src/docs/plan.md`; implementation planning docs live in `/docs` and should be kept in sync with product decisions.
- Do not implement multiple roadmap phases in one pass; follow `/docs/01-roadmap.md` and `/docs/08-task-breakdown.md`.

## Commands

- Package manager is npm; use `package-lock.json` as the dependency source of truth.
- Dev server: `npm run dev`.
- Typecheck: `npm run check` runs `svelte-kit sync` before `svelte-check`.
- Lint: `npm run lint` runs Prettier check before ESLint.
- Unit/component tests: `npm run test:unit`; use `npm run test:unit -- --run` for a non-watch run.
- Full tests: `npm test` runs Vitest once, then Playwright e2e.
- E2E tests: `npm run test:e2e`; this installs Playwright browsers and `playwright.config.ts` builds then previews on port `4173`.

## Svelte/Tailwind Notes

- `vite.config.ts` forces Svelte runes mode for project files outside `node_modules`.
- Tailwind CSS is loaded through `@tailwindcss/vite` and `src/routes/layout.css`; there is no separate Tailwind config file currently.
- Prettier uses tabs, single quotes, no trailing commas, print width `100`, and the Svelte/Tailwind plugins.

## Product Constraints

- MVP uses local seed data first; no database, custom backend, auth, runtime Wikipedia calls, or Threlte implementation until earlier phases are stable.
- UI must consume normalized domain models, not raw Wikipedia/Wikimedia responses.
- External enrichment should be done by local scripts that write generated JSON consumed by the app.
- Every external entity should preserve `sourceUrl` when available and tolerate missing metadata.
- Keep 3D code isolated from regular UI code; the Threlte scene receives normalized data via props and does not fetch data directly.

## Existing Starter Files

- `src/routes/+page.svelte`, `src/routes/demo/**`, and `src/lib/vitest-examples/**` are starter/demo files, not product architecture.
